# Lessons Learned — v0.1-phase-3-impl

## Process-Level

1. **shopt -s inherit_errexit does NOT propagate set -e into bash functions invoked via if-conditional context** — The POSIX errexit-context rule overrides `inherit_errexit` for any function called inside an `if`-statement, `&&`-chain, or `||`-chain. This means `if ! _writeback_state; then` and `_writeback_state || ...` both suppress `set -e` inside `_writeback_state`, regardless of `shopt -s inherit_errexit`. Reviewers and implementers must NOT assume `inherit_errexit` is sufficient when a function is invoked via if-context. The production-grade pattern is explicit per-call `|| { sentinel=...; return 1; }` guards inside any function whose failure paths must surface to the caller.
   _Discovered: Pass 6 Fix Burst 6 (40de399), 2026-05-28. Root cause exposed by test 45 RED GATE in 9fe29ce transitioning FAIL → PASS after 40de399 applied explicit guards._

2. **Paper-fix detection works when test-writers have autonomy to commit initially-failing tests** — Pass 4 F-P4-O01 was "closed" at 5c8430a with `shopt -s inherit_errexit`. Pass 6 adversary surfaced the same issue as F-P6-C01. The paper-fix was only exposed when test-writer added a load-bearing yq-failure test (test 45) in 9fe29ce that initially FAILED — demonstrating the claimed fix was insufficient. The paper-fix was then actually closed at 40de399. Lesson: empirical RED GATE tests that start failing are more reliable closure evidence than implementer self-assessment. TD-VSDD-059 paper-fix detection protocol requires load-bearing tests precisely for this reason.
   _Discovered: Pass 6 (2026-05-28). Manifestation: 2-step closure required for a finding that was declared closed in Pass 4._

3. **Fresh-context audits should explicitly include narrative prose review (Description, AC text, Architecture Compliance Rules), not just enum/structural artifacts** — Postcondition-focused passes can leave Description prose unswept, creating partial-fix regression (S-7.01). The v1.3 → v1.4 burst correctly enumerated Postcondition 5 and its 4 writeback fields, but left BC-2.01.006 Description claiming "the skill fires the hook" intact. Seven consecutive passes missed this drift. Pass 8, a fresh-context audit, caught it by reading the Description prose against the actual BC-2.04.014 architecture. Lesson: adversary dispatch instructions should explicitly call out narrative sections (Description, Preconditions prose, Architecture Compliance Rules) as review surfaces alongside structured tables and enumerations.
   _Discovered: Pass 8 (2026-05-28). Root cause: partial-fix regression from v1.3→v1.4 burst. Closure: BC-2.01.006 v1.5 Description rewritten at 03a34d3._

4. **Reconciliation documents must sweep the WHOLE file in scope, not just named sub-fields** — BC-DIMENSION-RECONCILIATION.md §4 listed `plugins/brain-factory/templates/state-md-template.md` as a file where "frontmatter additions" would be made. The frontmatter was correctly updated with the six canonical dimension names. However, the body of the same file retained legacy dimension labels (Source Coverage, Wiki Completeness, Embedding Status, Brief Currency, Health, Convergence). Pass 9 fresh-context audit caught this partial-fix as F-P9-C04 (CRITICAL). The reconciliation document's scope statement said "frontmatter additions" — but the reconciliation was responsible for the file, not just one section of it. Production-grade rule: when a reconciliation document lists a file in scope, the reconciliation must sweep the WHOLE file, or explicitly enumerate which sections are intentionally out-of-scope. The orchestrator and PO closing the reconciliation must verify completeness across all sections of every in-scope file, not just the sections referenced in the reconciliation plan.
   _Discovered: Pass 9 (2026-05-28). Root cause: BC-DIMENSION-RECONCILIATION.md §4 partial-scope statement. Closure: state-md-template.md body rewritten at aad5374 (P9 fix burst 9)._

   **Refinement (Pass 10, 2026-05-29):** Lesson L4 applies recursively — even bursts that CITE L4 in their commit body must apply L4 to themselves. Pass-9 fix burst correctly cited L4 for the BC-DIMENSION-RECONCILIATION partial-fix on state-md-template body, but failed to apply L4 to its OWN sweeps: AC-009/Task #2 (adjacent story-body sections in STORY-004.md adjacent to AC-010/Task #3 swept by reconciliation), BC-2.04.014 Invariants section (adjacent to H1/Description/Postconditions swept by v1.5), VP-024 prose preamble (adjacent to test-code snippet already updated). Pass 10 adversary surfaced all four as IMPORTANT findings (F-P10-I01 through I04). Production-grade rule: when a fix burst sweeps file F with intent X, the fix burst must explicitly enumerate ALL sections of F that are in-scope for X and sweep every section before declaring closure. The intent "ALWAYS exits 0 contract" applies to ALL sections of BC-2.04.014 (including Invariants), and the intent "E-HEALTH-001 exit 2" applies to ALL sections of VP-024 (including prose preamble), not just the sections the implementer happened to look at.
   _Discovered: Pass 10 (2026-05-29). Root cause: Pass-9 fix burst cited L4 in commit message but applied section-scoped sweeps, not whole-file sweeps. Closure: STORY-004 AC-009+Task#2 (9cbcb22), BC-2.04.014 v1.6 Invariant 4 (ac9dc62), VP-024 prose preamble (658f712)._

5. **Cascade depth reflects slow-rotting spec drift; fresh-context findings should be closed mid-cascade per Rule 4 rather than deferred** — The BC-5.39.001 3-CLEAN protocol's true value is catching slow-rotting spec drift that earlier passes deprioritize or classify as "out of scope." Pass 11 surfaced findings (volatile line-number trace pin in AC-009, sibling-sweep miss in STORY-004 line 258, VP-024 Counterexample failure-mode ambiguity) that prior passes would have classified as edge-cases or deferred. All were real defects. The production-grade rule: in convergence cascades, treat fresh-context findings as authoritative regardless of "scope" classification. The scope decision should be made by orchestrator routing, NOT by skipping findings. Per CLAUDE.md production-grade default Rule 4 (AI-built defects = AI responsibility), the right disposition for spec drift surfaced mid-cascade is closure in-scope, not deferral. Exception: POL-14-governed findings (BC/story frontmatter `status: draft` pre-merge) are a well-defined false-positive class and should be recognized as such by adversary dispatch instructions, not closed mid-cascade — add explicit guidance to adversary dispatch template. Lesson 5 also highlights the value of the adversary dispatch "Already CLOSED" list: a growing list that includes POL-14 deferrals prevents false-positive recurrence without adding closure burden.
   _Discovered: Pass 11 cascade analysis (2026-05-29). Root cause: mixed slow-rotting spec drift (F-P11-01 through 04) + false-positive class (F-P11-05/06 POL-14) + historical-reference non-issue (F-P11-O01). Fix burst 11 (97630bd + b2f339c) closed actionable findings; POL-14 deferrals codified in adversary dispatch template for Pass 12._

6. **Cross-story spec collisions (BC-vs-story precedence) surface only when fresh-context audit derives the contract from authoritative sources rather than the story under review** — Pass 12 found that STORY-004 AC-005 specified a RED threshold of 200K tokens, while BC-2.16.002 specifies RED at 150K tokens / 3x ingest volume. The implementer had followed the story spec; the story had not been cross-checked against its referenced BC for contract semantics. Per CLAUDE.md Source-of-Truth Precedence rule 1: the BC is authoritative for contract semantics; the story (and any impl that followed the wrong story) must be brought into alignment. This is not a matter of scope — the BC defines the external-observable contract, and the story can only narrow or clarify (not contradict) the BC for implementation scope. Production-grade rule: when a story spec disagrees with its referenced BC on contract semantics (e.g., threshold values, exit codes, enum values), the BC wins. Implementer should verify story-vs-BC alignment during initial implementation review before any TDD begins. Adversary fresh-context cascade catches these collisions precisely because it reads the BC independently, not through the lens of the story under review.
   _Discovered: Pass 12 (2026-05-29). Root cause: STORY-004 AC-005 RED threshold 200K vs BC-2.16.002 150K/3x; story was authored before BC-2.16.002 was fully locked. Fix: STORY-004 AC-005 + Compliance Rule 4 + Task #3 + run.sh constant aligned to BC-2.16.002 at 624dc86+9be3702. Collateral closure: DI-004 (STORY-005 exit-code drift, same BC-vs-story class) closed in-scope at e8504ef per L5/Rule 4 application._

7. **Citation drift — BC Canonical Test Vector citations must reference actual `@test` declarations from the bats suite, not function names** — When a BC's Canonical Test Vectors table cites a function name (e.g., `brain_health_skill_run`) rather than an actual bats `@test` declaration name, the citation can become non-existent silently. No test ever fails because the BC's text is documentation, not a load-bearing assertion. The cited function name can be removed, renamed, or never exist, and no automated check will surface the gap until a fresh-context adversary reads the BC against the actual test file. Production-grade rule: BC Canonical Test Vector citations must reference the exact `@test` string (e.g., `@test "brain_health_skill AC001 produces six-dimensional output"`) from the bats suite. Verify by grep when introducing or updating citations. The distinction is subtle but critical: a _function name_ in bash source is an implementation detail; a _@test name_ in bats is the interface that assertions reference.
   _Discovered: Pass 13 (2026-05-29). Root cause: BC-2.01.006 Canonical Test Vectors table was authored with function names from the implementation rather than @test declaration names from the bats suite. Closure: BC-2.01.006 v1.6→v1.7 at 0bd4612 — all citations rewritten to actual @test names._

8. **Adversary tool dispatches must use absolute paths within the STORY worktree** — The `cd` prefix in an adversary dispatch prompt sets the shell CWD for bash commands but does NOT constrain Read/Grep absolute-path resolution in Claude Code tool calls. When the dispatch prompt includes `cd /Users/jmagady/Dev/brain-factory/.worktrees/STORY-004 &&` but then lists spec/story files as relative paths (e.g., `.factory/specs/behavioral-contracts/ss-01/BC-2.01.006.md`), the Read tool resolves those relative paths against the tool's own CWD — which may be the main project root rather than the worktree. The result is that the adversary reads `/Users/jmagady/Dev/brain-factory/.factory/...` (the factory-artifacts worktree, which may be stale on a story branch) instead of `/Users/jmagady/Dev/brain-factory/.worktrees/STORY-004/.factory/...` (the story branch's live checkout). In the Pass 14 case the factory-artifacts worktree was at stale commit f85dc89 (pre-STORY-004 era), causing 9 of 12 adversary findings to be BOGUS — the adversary was reading old file versions and finding issues that had already been closed in fix bursts 1 through 13. Production-grade rule: adversary dispatch prompts must list ALL reviewable artifacts as explicit absolute paths rooted at the worktree directory (e.g., `/Users/jmagady/Dev/brain-factory/.worktrees/STORY-004/.factory/specs/behavioral-contracts/ss-01/BC-2.01.006.md`). The dispatch template section in SESSION-HANDOFF.md §Adversary Dispatch Template must pre-list all artifact paths in absolute form. Never rely on `cd` prefix to anchor relative paths for adversary tool reads.
   _Discovered: Pass 14 (2026-05-29). Root cause: dispatch template used `cd /Users/jmagady/Dev/brain-factory/.worktrees/STORY-004 &&` prefix with relative `.factory/...` paths, causing adversary Read tools to fall through to stale factory-artifacts worktree at f85dc89. 9 of 12 Pass-14 findings were BOGUS. 3 real findings (F14-06, F14-11, F14-12) closed at 9b25092+d52e6f4. Dispatch template updated in SESSION-HANDOFF.md §Adversary Dispatch Template for Pass 15 to use explicit absolute worktree paths._

9. **Bats JSON-shape assertions must use `jq -e` structural extraction, not substring matching** — Bats tests asserting JSON output shape via substring matching (`[[ "$output" == *"FIELD"* ]]`) are NOT load-bearing on field names. They pass against any output containing the expected substring regardless of structural position, data type, or parent object. A test like `[[ "$output" == *"red_dimensions"* ]]` passes even if `red_dimensions` appears in an error message or comment string, not as a JSON field. This is a paper-fix-prone pattern per TD-VSDD-059: the assertion can be satisfied by coincidental output while the hook contract is completely violated. Production-grade rule: JSON-shape assertions must use `jq -e` (or equivalent structural extraction) to validate field presence, type, and value. For hook stdout conformance tests, extract the specific field path with jq and assert on the extracted value. Substring-only assertions are acceptable ONLY for non-JSON string output (e.g., plain-text help messages) where there is no structured schema to validate. The test-writer agent should default to `jq -e` for any operator-facing JSON contract and treat substring-only assertions as a red flag during adversarial review.
   _Discovered: Pass 15 (2026-05-29). Root cause: brain-health-check.bats assertions used `[[ "$output" == *"FIELD"* ]]` substring tests for BC-2.04.014 v1.6 hook output schema (systemMessage/additionalContext/unhealthy_state/red_dimensions). Pass-15 adversary flagged as F15-03 (MEDIUM, paper-fix risk per TD-VSDD-059). Closure: 7eed597 replaced with structural jq -e assertions (+10 tests, 22→32)._

10. **Convergence-pass observations on benign dead-code patterns are NOT blockers per BC-5.39.001 when they don't alter contract behavior** — Pass 20 OBS-01 observed a dead-code pattern in brain-health-check.sh: the F17-04 fallback's `*":"*` branch is unreachable because the `case` statement's prior branch (`"fatal"*`) already matches all `fatal:*` lines. This is a code-clarity observation, not a contract violation. The hook behavior, output schema, and all contract invariants are unaffected. Production-grade rule: code-clarity observations (dead code, redundant conditions, cosmetic branch ordering) are recorded as OBS in the pass report but do NOT reset the BC-5.39.001 streak unless they expose a contract violation or introduce a real defect risk. The distinction: an observation that says "this code path never executes but the contract is intact" is informational; an observation that says "this code path may execute and produce wrong output" is a contract violation (IMPORTANT or CRITICAL). Adversary agents should apply this classification discipline explicitly in Pass N reports — OBS with "non-blocking" rationale prevent false streak resets.
   _Discovered: Pass 20 (2026-05-29). OBS-01: F17-04 fallback `*":"*` branch in brain-health-check.sh unreachable behind `"fatal"*` branch — benign dead code, no contract impact. Pass 20 classified PASS; BC-5.39.001 streak advanced 2/3 → 3/3 → CONVERGED._

11. **Harness `TaskCreate`/`TaskUpdate` tasks are ephemeral and LOST on session reset — orchestrator must mirror working task state to `.factory/TASK-LIST.md` after every meaningful burst** — The Claude Code harness `TaskCreate`/`TaskUpdate` tasks live only in the current orchestrator session's memory. They do NOT persist to disk and are LOST on session reset. When a fresh orchestrator session starts, harness task state from the prior session is completely invisible — the new session has no record of which stories are pending, which DIs are active, which user decisions were made, or what the next-action is. `.factory/TASK-LIST.md` is declared as `canonical_task_list` in STATE.md frontmatter. A fresh session reading STATE.md + SESSION-HANDOFF.md + TASK-LIST.md (in that order) must be able to resume the cycle with zero context loss. Production-grade rule: after every meaningful burst (story delivery, wave gate, user decision recorded, DI state change, cycle lesson added), the orchestrator mirrors the working task state to TASK-LIST.md as a single factory-artifacts commit. The ephemeral harness task state must NEVER be the only record of the pipeline's current working plan. This lesson was identified after STORY-004 LOCAL cascade (20 passes, 17 fix bursts, ~30 commits) — the harness tracked detailed task state throughout, but TASK-LIST.md remained frozen at Phase 2 Step G CONVERGED (2026-05-19), invisible to any fresh-context session resume.
   _Discovered: post-STORY-004 delivery (2026-05-29). Root cause: no discipline existed for mirroring harness task state to TASK-LIST.md after each burst. TASK-LIST.md was frozen from 2026-05-19 despite Wave 4 delivering 3/4 stories across multiple sessions. Closure: TASK-LIST.md rewritten to current state; SESSION-HANDOFF.md §Step 1 updated to list TASK-LIST.md as mandatory third read; this lesson codified._

12. **Cascade paper-fix layers descend through architectural strata — each pass exposes a deeper layer that prior passes and humans institutionally missed** — The STORY-015 adversarial cascade found paper-fixes at successive architectural strata: Pass 1 (assertion strength — substring matching vs structural validation of BC compliance), Pass 2 (implementation envelope — nested-function code path vs top-level code path), Pass 4 (fixture content — singular vs plural category names in test fixtures), Pass 5 (test injection mechanism — shell-quoting brittleness causing silent test pass on malformed input). Each layer appeared only after the prior layer was fixed; no single pass saw all layers. The pattern: a fresh-context adversary with strict BC reading independently re-derives contract requirements from first principles — this catches drift between BC text, test mechanism, fixture content, and implementation that prior passes (which accumulated context) stopped questioning. BC-5.39.001 3-CLEAN protocol IS the layered-defense mechanism; the cascade depth is a feature of rigorous convergence, not a process failure. Accept cascade depth as evidence the protocol is working. If this pattern recurs across multiple stories (STORY-004 had similar depth), codify as a sweep rule: at fix-burst time, explicitly verify the BC text → test mechanism → fixture content → implementation chain end-to-end, not just the layer the adversary flagged.
    _Discovered: STORY-015 LOCAL cascade Passes 1-5 + fix bursts 1-5 (2026-05-30). Cascade: 8 passes, 5 fix bursts, BC-5.39.001 3-CLEAN at Passes 6+7+8. Merge PR #19 at 20bedb7._

13. **Upstream vsdd-factory compute-input-hash tool has an awk-termination bug — state-manager must use manual workaround for story input-hash population** — The `vsdd-factory:compute-input-hash` tool's awk rule `/^[^ -]/` fires after `sub(/^  *- */, "")` modifies `$0`, causing awk to terminate after processing the first input file in the list. The result is a hash computed from only the first input, not the full concatenation. Workaround: state-manager manually computes the correct hash via a CONCAT loop (`for f in ...; do cat "$f"; done | sha256sum`). This affects every story's input-hash population during STORY-NNN closure bursts. Future input-hash bursts MUST use the manual workaround OR wait for the upstream drbothen/vsdd-factory fix to land. Do NOT use the tool in its current form for multi-input hash computation; it will silently produce an incorrect (single-file) hash. An upstream issue should be filed against drbothen/vsdd-factory for the awk-termination bug fix.
    _Discovered: post-STORY-004 delivery (closes F-PASS02-O02 process-gap). Affects: all story input-hash fields populated via compute-input-hash tool. Workaround applied for STORY-004 and STORY-015 closure bursts._

14. **Worktree mount must respect project standing directive — devops-engineer pre-flight skills must check for project-specific `worktree_layout_note` overrides BEFORE auto-mounting** — brain-factory's STATE.md frontmatter declares `worktree_layout_note: .factory/ is a regular directory tracked on main with factory(...) conventional commits per SESSION-HANDOFF §10 standing directive (intentional pre-v0.1 state; NOT a regression)`. The devops-engineer `factory-worktree-health` auto-repair at session start mounted `.factory/` as a worktree on the `factory-artifacts` branch — the generic vsdd-factory default. This created a STALE READ surface: when the adversary Gate 3 re-verify subsequently read `.factory/...` files, it resolved against the factory-artifacts worktree (at commit af6f004, Wave 3 era), NOT the live main branch (at commit 1622696 with the Gate 3 fix burst). The result was a false-negative Gate 3 re-verify FAIL — the adversary concluded the fix was absent when it was in fact present on main. Fix: `git worktree remove .factory --force` + `git checkout main` in the parent worktree restores the correct layout. Lesson: pre-flight health scripts must read the project's `worktree_layout_note` field BEFORE applying any default worktree setup. For brain-factory, the standing directive is `.factory/ on main` — do NOT mount as a worktree on `factory-artifacts` without explicit human authorization. Codify as a pre-flight check: if `worktree_layout_note` exists in STATE.md frontmatter, apply it and skip the default factory-artifacts auto-mount.
   _Discovered: Wave 4 Gate 3 re-verify session (2026-05-30). Root cause: devops-engineer factory-worktree-health auto-repair applied generic vsdd-factory default, overriding brain-factory's standing directive. False-negative revert: adversary read stale factory-artifacts content at af6f004 instead of live main at 1622696. Fix: worktree removed, main checkout restored._

15. **Wave-perimeter adversary catches state-vs-implementation drift that per-story passes cannot see** — STORY-017 and STORY-032 BCs were claimed promoted in STATE.md narrative (via POL-14 language) but the BC files on disk still showed `status: draft`. Per-story Step 4.5 adversary cannot detect this class of drift because POL-14 promotion happens AT merge, after Step 4.5 completes — the promotion is a post-merge bookkeeping operation that the per-story cascade has no visibility into. The Wave 4 Gate 3 adversary, reading the full wave diff with a fresh context, caught BOTH the narrative-vs-file divergence (C01) AND missing demo evidence for STORY-017/032 (H01) in the same pass. This validates the three-perimeter model (BC-5.39.002 PC7): per-story LOCAL cascade verifies story-level contract conformance; PR-level adversary verifies pre-merge diff quality; wave-gate adversary verifies wave-level integration and state consistency. Each perimeter catches a different defect class. The wave-gate perimeter specifically detects: (a) POL-14 promotion gaps (narrative claims active, file says draft), (b) cross-story demo evidence gaps, (c) state-vs-spec divergence that only becomes visible when the whole wave diff is read together. Wave-gate adversary passes are NOT redundant with per-story passes — they have structural information asymmetry that enables a different defect class detection.
   _Discovered: Wave 4 Gate 3 (2026-05-30). C01: BC-2.02.002/003/005 + BC-2.12.001/002 still status:draft despite STATE.md claiming promotion. H01: STORY-017/032 missing demo evidence files. Both caught by wave-gate adversary pass reading full wave diff. Fix burst 1622696 (5 BCs promoted) + PR #20 21533b0 (15 demo files). Lessons validates three-perimeter model (BC-5.39.002 PC7)._

16. **When a project diverges from canonical vsdd-factory patterns, document the divergence explicitly + plan reconciliation timeline** — brain-factory's pre-v0.1 standing directive (STATE.md frontmatter `worktree_layout_note`) placed `.factory/` as a tracked directory on main rather than mounting an orphan factory-artifacts branch. This was an intentional pre-v0.1 decision (minimal git complexity early) — but it accumulated 68 factory(state) commits on main and confused multiple agents during Wave 4 gate when devops-engineer auto-mounted `.factory/` as a worktree (overriding the standing directive) and adversary Gate 3 re-verify read stale factory-artifacts content (false-negative FAIL). Production-grade rule: when diverging from canonical patterns, STATE.md must (a) document the divergence with explicit rationale, (b) define a reconciliation trigger (e.g., "before Wave N" or "before v1.0 ship"), and (c) maintain awareness in agent dispatches so adversary/devops-engineer don't silently auto-correct under canonical assumptions. Reconciliation executed 2026-05-30 at Wave 4 end per user authorization; factory state migrated from main-tracked to factory-artifacts orphan mount.
   _Discovered: Wave 4 Gate 3 false-negative (2026-05-30). Root cause: divergence-without-explicit-reconciliation-plan plus agent auto-correction (devops-engineer factory-worktree-health pre-flight mounted .factory/ as worktree without checking standing directive). Closure: full reconciliation executed end of Wave 4; canonical topology restored. D-038 decision row documents reconciliation steps._
